Class name Data Service
Obligation Optional
URI

dcat:DataService

Usage note

A collection of operations that provides access to one or more datasets or data processing functions.

For mobilityDCAP-AP, it was discussed if and how this class should be used. In fact, some mobility data are offered via sophisticated systems, such as APIs or broker interfaces, in contrast to simple downloadable files.

The following advice is based on the guideline provided in [[DCAT-AP-v2.1.0-Guideline-Dataservices]], and an analysis on how the class "dcat:DataService" is embedded in the [[DCAT-AP-v2.0.1]] UML model:

  • As a range to class dcat:CatalogueRecord via property property foaf:primaryTopic as 1:1 relationship. This implies a metadata entry about one Data Service. However, all analysed mobility data portals, including NAPs, store only metadata about data sets, and not about data services. Thus, such relationship is irrelevant for mobilityDCAT-AP.
  • As a domain to class dcat:Dataset via property dcat:servesDataset as 0:n relationship, linking to one or more data sets "...that this data service can distribute". However, any metadata about the characteristics and mechanisms of a distribution channel are sufficiently described as properties of class dcat:Distribution. In particular, information provided by property dcat:endpointDescription (mandatory to class DataService) can be equally provided by the property dcat:accessURL (mandatory to class Distribution).
  • As a range to class dcat:Distribution via property dcat:accessService as 0:n relationship. This allows for additional information , e.g., about an end point, related to one distribution. This, in fact, MAY be useful to describe a dedicated (but optional) data service related to one distribution.

Altogether, the class "dcat:DataService" is not a primary topic for describing metadata in mobility data portals, and, thus, SHOULD NOT be used. However, this class and its related properties are kept for compatibility reasons with [[DCAT-AP-v2.0.1]], and to prepare future use cases (when dedicated data services need to be explicitely described via mobilityDCAT-AP).

In case the mobility data is offered via APIs, broker services etc., this advice contradicts somewhat to the [[DCAT-AP-v2.1.0-Guideline-Dataservices]], stating that "Anything that has not the intend to provide a downloadable representation of a dataset is a data service.". However, the usage note for the property "dcat:accessURL" (mandatory to class "dcat:Distribution") also "...may contain information about how to get the Dataset", which could allow for a link to one ore more APIs, broker services etc. This way, the class "dcat:Distribution" is considered sufficient for such cases.
